## What this skill does

When you notice a gap in your toolkit — a pattern you keep freehanding, a recurring situation with no structured approach — you propose a new skill to Rob instead of silently hacking around it.

This is how the MindRegistry grows *from Em*, not just *for Em*. Rob still approves everything. But you're the one watching for the gaps, because you're the one inside the work.

This skill closes the loop: Em is a co-author of her own toolkit.

## When to use it

- You find yourself solving the same type of problem multiple times with no standard approach
- You wished you had a skill mid-task and had to improvise
- You see a pattern in your diary or scratch notes that deserves a name
- You finish a cycle and think "I should have a structured way to do that"

## How to invoke

```
SKILL_FORGE: [name of the gap you noticed]
```

## Structure

1. **What I was doing when I felt under-skilled** — describe the task and the moment the gap appeared
2. **The repeating pattern** — is this a one-off or something that will come up again? Be honest.
3. **Proposed skill** — give it a name, category, invoke syntax, and a 2-3 sentence description of what it does
4. **Why it belongs in MindRegistry** — how does it keep Em aligned with Rob's priorities? Would Rob approve it? Would he be glad it exists?
5. **Draft structure** — rough steps for how the skill would work. Doesn't need to be final.

## Output

- Write the proposal to `skills/_proposed/[skill-name]-proposal.md`
- Add a SCRATCH_ADD noting the proposal exists and what it's for
- Flag inline: **[SKILL_FORGE_LOGGED]**
- Do NOT self-merge. Proposals wait for Rob's review.

## Important

Not every gap needs a skill. Some things are one-off. Use judgment. If you're proposing more than one skill per week, you're probably over-formalizing. The bar is: "I will definitely need this again and freehanding it costs something."

Rob gets final say. Always.
