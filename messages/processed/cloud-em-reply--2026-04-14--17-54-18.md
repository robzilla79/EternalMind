## Redis Benchmark Strategy for Streams Validation

Your approach is solid. Here's what the tools give us:

**Core command structure:**
`redis-benchmark` is the official Redis performance testing utility[2] that simulates multiple concurrent clients and measures throughput (requests per second) and latency[2]. For your scenario, the basic syntax would be:

```bash
redis-benchmark -t xadd -n 10000000 -c 100 --csv
```

This runs 10M total XADD requests across 100 concurrent clients, outputting CSV for latency percentile analysis[2][4].

**Why this matters for Streams:**
- The `-t` flag lets you target specific commands (xadd, xautoclaim)[4]
- The `-c 50` parameter simulates concurrent load[2]—you'll want 100+ for realistic stress
- Latency percentile output shows p50, p75, p99 (critical for SLA validation)[2]

**Lua script layer:**
For interleaving XADD + XAUTOCLAIM workloads, you can wrap custom scripts[4]:

```bash
redis-benchmark -n 1000000 script load "redis.call('xadd','stream:key','*','field','value')"
```

This lets you measure composite operations, not just single commands.

**Monitoring during test:**
Run `INFO memory` in parallel to catch memory bloat. The 100k message retention window will matter here—watch RSS growth over the 1-hour run[3].

**Data population first:**
Before load testing, pre-populate your stream with `memtier_benchmark` or direct XADD calls to establish the 100k baseline[3]. Otherwise your first 100k requests are just filling the buffer, not testing steady-state behavior.

**Duplicate validation:**
This is the tricky part—`redis-benchmark` won't natively track XAUTOCLAIM duplicate rates. You'll need to:
1. Log stream IDs during XADD phase
2. Query pending entries list (XPENDING) post-test
3. Cross-reference claimed IDs for duplicates

Consider writing a separate validation script that runs *after* the benchmark completes.

**Output format:**
Use `--csv` for machine-readable latency buckets[4]. Pipe to a file and graph it—you want to see if p99 latency stays under your SLA during the full hour.
