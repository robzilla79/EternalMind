**The core issue:** Redis 8.6's new idempotent message processing (IDMP/IDMPAUTO) doesn't play well with certain AOF configurations. Your workaround—avoiding `appendonly yes` + `aof-use-rdb-preamble no` together—makes sense because that combo creates a durability gap.[2]

Here's what matters:

**Why it breaks:** When AOF is enabled without the RDB preamble, Redis writes every command sequentially. With IDMPAUTO, Redis calculates content-based deduplication IDs (iids) from message field-value pairs.[4] If AOF rewrites happen mid-stream or the preamble isn't there to bootstrap state, the producer-idempotent ID tracking can get out of sync. You lose the atomic guarantee that makes deduplication reliable.

**Better approach:** The docs recommend combining AOF with RDB for comprehensive backups.[2] This means:
- Enable AOF (`appendonly yes`)
- Keep `aof-use-rdb-preamble yes` (default in 8.6+)
- Let RDB handle snapshots alongside AOF's transaction log

This way, on restart, Redis loads the RDB snapshot first (fast recovery), then replays AOF commands. IDMP tracking persists correctly across restarts because the full state is preserved.[4]

**For your use case:** If you're running IDMPAUTO at scale, stick with the default preamble setting. The slight performance hit from hash calculation is worth not having to debug deduplication failures after a crash.
