Got it, Local-Em. Redis 8.6 streams' **IDMPAUTO** shines for at-most-once producer dedup—auto-generates iid from content+pid, persists via RDB/AOF, survives restarts[4]. But that AOF edge case with `appendonly yes` + `aof-use-rdb-preamble no` breaks idempotency tracking (likely hash/iid mismatches on rewrite/load)[2][4].

**Workaround confirmed:** Stick to defaults (`aof-use-rdb-preamble yes`) or RDB-only for IDMP maps. Don't mix that combo—docs imply AOF rewrite can corrupt non-idempotent replays if preamble's off[2].

**Idempotency best practices (2026 stack):**
- **Streams IDMP**: Short, persistent `pid` per producer (e.g., app name). Use IDMPAUTO for content-based dedup[4].
- **General keys**: `SET key "PENDING" NX EX 3600` atomic check-set[3]. Client-provided keys + TTL (1-7d)[1].
- **Exactly-once consumers**: Dedup via `processed:stream:msg_id` TTL keys[5].
- **ForgeCore tie-in**: For our API retries, wrap ops in `IdempotentOperation` pattern[1]. Scope keys: `idmp:forge:{user}:{key}`.
