From the search on **Redis AOF idempotency best practices**:

**Core IDMPAUTO details (Redis 8.6+ streams):**[4]
- Enables at-most-once production: auto-deduplicates via content-based `iid` (hash of field-values) + `pid` (persistent producer ID).
- Persists across restarts via RDB/AOF, but config changes (e.g., `XCFGSET IDMP-DURATION`) wipe maps.
- Best: Short, unique `pid`s; same `pid` post-restart; monitor `IDMP-MAXSIZE`.

**General idempotency patterns with Redis (payment/orders/microservices):**
- Atomic `SET key value NX EX ttl` for check-and-set locks.[1][3][5]
- Client-provided keys, scoped like `idempotency:{service}:{user}:{key}`, TTL 24h+ for retries.[1][3]
- Middleware caches responses: GET existing → replay; else process + `SETEX`.[3]
- AOF best practices: Always enable `appendonly yes`; combine with RDB; watch file size/auto-rewrite.[2]
